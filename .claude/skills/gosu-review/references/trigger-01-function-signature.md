# Trigger 01: 함수 시그니처 (`func`)

> 이 파일은 gosu-review SKILL의 트리거 01 레퍼런스다.
> 메인 SKILL.md의 워크플로우 3단계에서 이 트리거가 발동하면 읽는다.

## 발동 조건

함수/메서드 선언 (`func Name(...) ...`, `func (r Recv) Name(...) ...`)

## 검사 절차

1. 시그니처에서 파라미터/반환/리시버 확인
2. 반환 타입에 인터페이스 또는 `any`가 있으면 → Rule 7, 8, 44 추가 검사
3. Rule 44는 시그니처만으로 부족 — **함수 본문의 모든 `return` 문**을 추가로 본다
4. 파라미터에 파일 경로(`string`)가 있고 내부에서 `os.Open` 등을 호출하면 → Rule 45
5. 파라미터가 4개 이상이거나 nullable(`*T`, `bool`) 다수면 → Rule 11

## 룰

### Rule 7 — 인터페이스 반환 지양
- 핵심 질문: 함수가 인터페이스를 반환하고 있는가? 호출처에서 구체 타입이 더 적절하지 않은가?
- 원칙: **입력은 인터페이스(받아들이는 측은 유연하게), 출력은 구체 타입(주는 측은 명확하게)**
- 예외: `error`, 표준 라이브러리 패턴(`io.Reader` 반환 등), 다중 구현체가 실제로 있는 경우

### Rule 44 — nil 인터페이스 반환 함정
- 핵심 질문: 인터페이스를 반환하는 함수가 nil 포인터를 그대로 반환하지 않는가?
- 문제: `var p *MyError; return p` → 호출 측에서 `err != nil`이 **참**이 됨 (nil 포인터를 감싼 비-nil 인터페이스)
- 검사 방법: 인터페이스 반환 함수의 모든 return을 보고, 구체 포인터를 그대로 반환하는 경로가 있는지 확인
- 수정: 명시적으로 `return nil` 하거나, 분기에서 인터페이스 변수에 대입 후 nil 검사

### Rule 8 — any 남용
- 핵심 질문: 파라미터/반환에 `any` (또는 `interface{}`)를 쓰는데 정말 필요한가?
- 정당한 경우: `json.Marshal`처럼 임의 타입 수용이 본질인 경우
- 위반: 단순히 "여러 타입을 받고 싶어서" — 컴파일 타임 안정성을 잃음. 제네릭 또는 다중 시그니처 검토

### Rule 45 — io.Reader 활용
- 핵심 질문: 파일 이름(`string`)을 받아 내부에서 `os.Open` 하는가?
- 권장: `io.Reader`를 받아 호출 측이 파일/메모리/네트워크 어디서든 공급할 수 있게
- 이점: 테스트 용이성(`strings.NewReader`), 재사용성, 자원 관리 책임 분리

### Rule 11 — 함수형 옵션 패턴
- 핵심 질문: 옵션성 파라미터가 많거나 향후 추가 가능성이 큰가?
- 휴리스틱: 파라미터 4개+, nullable/zero-value 의미 의존, "기본값으로 두고 싶은 옵션" 다수
- 적용: `func New(required T, opts ...Option) *X` 패턴
- 예외: 항상 모든 파라미터가 필요한 함수, 단순 헬퍼

### Rule 42 — 리시버 타입 결정 (값 vs 포인터)
- 핵심 질문: 리시버가 값/포인터 중 무엇이고 그 선택이 타당한가?
- 기준: (1) 변경 필요 → 포인터, (2) 큰 구조체 → 포인터(복제 비용), (3) 동시성 안전성 의도 → 값, (4) `sync.Mutex` 등 복제 금지 필드 포함 → 포인터
- 일관성: 같은 타입의 메서드들이 값/포인터를 섞어 쓰면 안 됨
- 애매하면 포인터

### Rule 43 — 기명 반환값
- 핵심 질문: 기명 반환값을 쓸 만한 이유가 있는가?
- 정당: 같은 타입의 반환값이 여러 개여서 가독성 향상, `defer`에서 반환값 수정 필요
- 위험: zero-value 의존 의도 불명확, 섀도잉 버그(`err :=`로 가리는 등)
- 단순한 함수에 기명 반환값을 굳이 쓰지 말 것
