# Trigger 07: range 순회 (`for ... range ...`)

> 이 파일은 gosu-review SKILL의 트리거 07 레퍼런스다.
> 메인 SKILL.md의 워크플로우 3단계에서 이 트리거가 발동하면 읽는다.

## 발동 조건

모든 `for ... range ...` 구문

## 검사 절차

1. range 대상의 정적 타입 확인
2. 루프 본문이 **값 변수를 수정**하거나 **필드를 대입**하고 있는가? → Rule 31
3. map → Rule 33 검토 (순서 의존 코드 탐지)
4. string → Rule 36, 37 검토 (룬/바이트 혼동)
5. **루프 본문에서 range 대상을 수정**하는 코드가 있는가? → Rule 32

## 룰

### Rule 31 — range 값 원소는 복제본
- 핵심 질문: `for _, v := range s`의 `v`가 **원소의 복제본**이라는 점을 의식했는가?
- 함정 — 구조체 필드 수정이 반영 안 됨:
  ```go
  type User struct{ Name string }
  users := []User{{"Alice"}, {"Bob"}}
  for _, u := range users {
      u.Name = "X"  // 복제본이 수정됨 — 원본 변경 안 됨
  }
  // users는 그대로 [{"Alice"}, {"Bob"}]
  ```
- 수정 방법:
  - **인덱스로 접근**: `for i := range users { users[i].Name = "X" }`
  - **포인터 슬라이스**: `[]*User`로 바꾸고 `for _, u := range users { u.Name = "X" }`
  - **일반 for 루프**: `for i := 0; i < len(users); i++ { ... }`
- 포인터 원소(`[]*T`)는 복제본도 같은 객체를 가리키므로 필드 수정이 원본에 반영됨 — 문제 없음
- 큰 구조체의 읽기 전용 순회에서도 값 복제 비용이 큼 → 포인터 슬라이스 또는 `for i := range &arr` 고려 (Rule 32 함정 2와 연계)

### Rule 32 — range 표현식 단일 평가
- 핵심 질문: `range` 대상 표현식이 루프 시작 시 **단 한 번만 평가**된다는 점을 의식했는가?
- 함정 1 — 슬라이스에 append하며 순회:
  ```go
  s := []int{0, 1, 2}
  for range s {
      s = append(s, 10)  // 무한 루프 X — 원본 s의 스냅샷으로 순회됨 → 3회만
  }
  ```
  루프는 초기 길이/백킹 배열을 기준으로 진행. 의도대로 동작할 수도, 예상과 다를 수도 있음
- 함정 2 — 큰 배열 값 복제:
  ```go
  var arr [1<<20]int
  for i := range arr { ... }   // 배열 전체가 한 번 복제됨! (슬라이스/포인터로 바꾸자)
  ```
  배열(슬라이스 아님)을 range하면 배열 전체 값 복제가 일어남. `for i := range &arr`처럼 포인터로 받거나 슬라이스로 변환
- 함정 3 — 채널 재할당:
  ```go
  ch := makeCh()
  for v := range ch {
      ch = anotherCh()  // 반영 안 됨 — 첫 ch에서 계속 읽음
  }
  ```
- 가이드: range 대상 변수의 재할당/대량 복제를 의식하고, 배열은 가능하면 슬라이스/포인터로 처리

### Rule 33 — Map 순회 특성
- 핵심 질문: range 대상이 map인데 다음 중 하나라도 의존하지 않는가?
  - (1) 키가 정렬되어 있음
  - (2) 추가 순서대로 나옴
  - (3) 매 반복마다 같은 순서
  - (4) 반복 중 추가된 원소가 항상/절대 포함됨
- Go의 map은 모든 항목에 대해 보장 없음
- 정렬된 순회가 필요하면 키를 슬라이스로 뽑아 정렬 후 순회

### Rule 36 — 룬과 바이트
- 핵심 질문: 문자열을 바이트 단위로 인덱싱/슬라이싱하면서 다국어를 깨뜨릴 위험이 없는가?
- 룬은 가변 길이(1~4바이트). `s[i]`는 바이트 접근
- 다국어 처리 시 `utf8.DecodeRuneInString`, `[]rune(s)`, `unicode/utf8` 활용

### Rule 37 — 문자열 range
- 핵심 질문: `for i, r := range s`에서 `i`가 **룬의 시작 바이트 인덱스**임을 의식했는가?
- 함정: `i`가 0, 1, 2, 3...이 아님 — 다국어 문자열에서는 건너뜀
- 룬 인덱스로 직접 접근하려면: `runes := []rune(s); runes[k]`

## 출력 시 주의사항

map 순회에서 Rule 33을 보고할 때는 "실제로 순서에 의존하는 코드가 맞는지" 근거를 명확히 밝힐 것. 단순 순회는 위반이 아니다.
