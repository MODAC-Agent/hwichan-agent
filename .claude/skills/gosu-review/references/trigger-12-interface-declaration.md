# Trigger 12: 인터페이스 선언 (`type ... interface`)

> 이 파일은 gosu-review SKILL의 트리거 12 레퍼런스다.
> 메인 SKILL.md의 워크플로우 3단계에서 이 트리거가 발동하면 읽는다.

## 발동 조건

`type Name interface { ... }` 선언

## 검사 절차

1. 이 인터페이스의 구현체 수 (1개? 다수?)
2. 인터페이스가 정의된 패키지 vs 사용/구현 패키지

## 룰

### Rule 5 — 인터페이스 선제 생성 금지
- 핵심 질문: "필요할지도 모른다"는 이유로 미리 만든 인터페이스인가?
- Go 철학: **추상화는 발견하는 것이지 만들어내는 것이 아니다**
- 트리거: 구현체가 단 1개, 테스트에서 모킹도 안 함, 명시적 다형성 요구 없음 → 인터페이스 제거 검토
- 예외: 외부 의존성 경계 (DB, HTTP 클라이언트 등 테스트에서 교체 필요)

### Rule 6 — 인터페이스 위치 (클라이언트 측 정의)
- 핵심 질문: 인터페이스가 구현체 패키지에 함께 선언되어 있는가?
- 권장: **사용자(클라이언트) 패키지**에 두기 — 사용자가 필요한 메서드만 정의
- 효과: 구현체 패키지가 인터페이스를 모름 → 결합도 감소, 작은 인터페이스 자연 유도
- 예외: 표준 라이브러리 패턴 (`io.Reader`처럼 매우 광범위하게 쓰이는 것)

## 출력 시 주의사항

Rule 5, 6은 패키지 구조 정보를 요구한다. 변경 파일 하나만 보면 구현체 수를 알 수 없을 수 있으니, 근거 란에 "이 파일에서는 확인 불가, 프로젝트 차원 검토 권장" 같은 단서를 명시할 것.
