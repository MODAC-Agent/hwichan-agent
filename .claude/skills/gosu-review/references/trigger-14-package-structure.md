# Trigger 14: 패키지/디렉토리 구조

> 이 파일은 gosu-review SKILL의 트리거 14 레퍼런스다.
> 메인 SKILL.md의 워크플로우 3단계에서 이 트리거가 발동하면 읽는다.
>
> **이 트리거는 항상 메인 에이전트가 직접 처리한다.** 파일 단위 서브에이전트에게 분배하지 말 것 — 전역 시야가 필요하다.

## 발동 조건

모든 `package` 선언, 디렉토리 이름

## 검사 절차

1. 모든 `package` 선언과 디렉토리명을 모아 본다
2. 모호 이름 패턴 매칭
3. 새 프로젝트면 표준 레이아웃 참고 권장

## 룰

### Rule 12 — 프로젝트 레이아웃
- 핵심 질문: 디렉토리 구조가 표준 레이아웃(예: golang-standards/project-layout)이나 도메인 단위로 명확한가?
- 새 프로젝트에 한해 권장 — 기존 프로젝트를 무리하게 재배치하지는 말 것
- `cmd/`, `internal/`, `pkg/` 등의 의미를 일관되게 사용

### Rule 13 — 패키지 네이밍
- 핵심 질문: `common`, `util`, `shared`, `helper`, `misc` 같은 책임이 드러나지 않는 이름이 있는가?
- 문제: "어떤 책임이든 들어갈 수 있는 폴더" → 응집도 저하
- 권장: 책임 단위로 분리 (`auth`, `payment`, `pricing`)
- 함수/타입 이름과 패키지 이름의 중복 회피 (`user.User` 대신 `user.Account` 등)

## 출력 시 주의사항

이 트리거의 발견은 **파일별이 아닌 프로젝트 단위 섹션**으로 리포트 앞부분에 별도 배치하는 게 자연스럽다. 리포트 포맷에서 "## 프로젝트 구조" 섹션을 추가할 것.
