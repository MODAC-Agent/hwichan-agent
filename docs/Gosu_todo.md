## 계획

먼저 책으로 학습 후, AI에게 라우팅 하기 쉬운 목차를 고민하며 구성한다. 실패 패턴 감지 후 참조하는 방식을 현재 생각중이다. 적당한 분류는 정확도에 더 도움될 것이다.

## 간단 정리

### 코드와 프로젝트 구성

1. 가려진 변수를 제거하면 엉뚱한 변수를 참조하거나 코드를 잘못 해석하는 실수를 방지할 수 있다.
2. 중첩 수준을 깊지 않게 유지하고 정상 경로는 항상 왼쪽으로 정렬하면 코드에 대한 멘탈 모델을 이해하기 쉽게 만들 수 있다.
3. 변수를 초기화할 때 init함수는 에러를 처리하고 상태를 관리하며 테스팅을 수항해는 데 한계가 있다는 사실을 명심한다. 웬만하면 초기화는 별도함수를 이용하여 처리한다.
4. 게터와 세터를 강제로 사용하는 것은 고 언어의 관례가 아니다. 실용적인 관점으로 접근하여 효율성과 특정 관례 준수 사이의 적절한 균형점을 찾는 것이 중요하다.
5. 추상환느 찾아내야지 새로 만들면 안 된다. 쓸데없는 복잡도가 증가하지 않도록 인터페이스는 미리 만들어두지 말고 필요한 상황이 발생하거나 최소한 올바른 추상화를 제시할 수 있을 때 만든다.
6. 인터페이스를 클라이언트 측에 두면 불필요한 추상화를 줄일 수 있다.
7. 유연성 관점에서만 보면, 함수에서 인터페이스보다는 구체적인 구현을 리턴하는 것이 바람직한 경우가 많다. 반대로 함수는 대부분 인터페이스르 받을 수 있어야 한다.
8. json.Marshal 등과 같은 모든 유형의 타입을 인수로 허용하거나 리턴할 때만 any를 사용한다. 나머지 경우 any만으로는 특별히 의미 있는 정보를 제공할 수 없으며 호출자가 임의의 데이터 타입으로
   메서드를 호출할 수 있게 만들면 컴파일 시간에 문제가 발생할 수 있다.
9. 제네릭 타입과 매개변수를 이용하면 보일러플레이트 코드를 작성할 필요 없이 특정 요소나 동작을 떼어낼 수 있다. 하지만 타입 매개변수를 섣불리 사용하지 말고 꼭 필요한 상황에만 사용한다. 그렇지 않으면 쓸데없이
   추상화만 늘어나고 복잡도가 늘어난다.
10. 타입 임베딩도 보일러플레이트 코드를 제거하는 데 도움이 된다. 하지만 일부 필드를 공개하면 안 되는 상황에서 가시성 관련 문제가 발생할 가능성도 있다.
11. 옵션을 편리하면서 API와 어울리는 방식으로 처리하려면 함수형 옵션 패턴을 적용한다.
12. Go 프로젝트를 처음 구조화하는 데는 project-layout과 같은 특정 레이아웃을 따르는 것이 도움이 된다. 특히 새로운 프로젝트를 표준화하는 기존 규칙을 원한다면 더욱 그렇다.
13. 이름 짓기 역시 애플리케이션 디자인에서 핵심 역할을 한다. 패키지를 common, util, shared 같은 이름으로 만들면 사용자에게 별로 도움이 안 된다. 이런 이름을 가진 패키지는 좀 더 의미 있고
    구체적인 이름으로 리팩터링 한다.
14. 변수와 패키지 사이에서 이름이 충돌해 코드를 해석하기 힘들어지거나 버그가 발생할 수 있으므로 각각 고유한 이름을 갖도록 하는 것이 좋다. 여의치 않다면 임포트 앨리어스를 이용하여 변수 이름과 패키지 이름을
    구분할 수 있도록 한정자를 변경하거나 더 좋은 이름으로 바꾼다.
15. 클라이언트와 코드 관리자가 코드를 쉽게 이해할 수 있도록 익스포트한 요소는 반드시 문서화한다.
16. 린터와 포맷터를 이용하여 코드 품질과 일관성을 높인다.

### 데이터 타입

17. 기존 코드를 읽을 때 0으로 시작하는 정수 리터럴은 8진수라는 사실을 명심한다. 0o를 앞에 붙이자.
18. 고 언어는 정수 오버플로나 언더플로를 걸러주지 않기 때문에 감지하는 코드는 직접 작성
19. 주어진 범위 이내로 부동 소수점 수를비교하도록 작성하면 코드 이식성 향상
20. 덧셈 뺄셈을 수행할 때는 자릿수가 비슷한 연산끼리 묶는 것이 정확도를 높인다. 곱셈 나눗셈 먼저 수행 후 덧셈 뺄셈
21. 슬라이스의 길이 용량 차이를 이해하자.
22. 슬라이스 생성 시 길이를 알고 있다면, 길이나 용량을 지정해서 초기화하는 것이 좋다. 할당 회수를 줄이고 성능을 높인다. 맵도 크기를 지정하면 좋다.
23. append할 때 복제나 슬라이스 표현식을 사용하면 서로 다른 두 함수가 동일한 내부 배열에 기반한 슬라이스를 사용하더라도 충돌이 발생하는 일을 피할 수 있다. 슬라이스를 다시 줄이고 싶을 때는 슬라이스를
    복제하는 방식을 사용하여 메모리 누수를 줄이자.
24. 내장 함수 copy를 활용해 슬라이스를 복제하면, 원소 개수가 짧은 길이의 슬라이스에 맞춰진다.
25. 포인터 슬라이스나 포인터 필드가 담긴 구조체를 다룰 때, 슬라이싱 연산으로 제외되는 원소를 nil로 설정해야 누수를 막는다.
26. encoding/json, reflect 패키지를 사용할 때와 같은 경우, nil과 empty 슬라이스 차이를 알아야 한다. nil은 할당이 없다.
27. 슬라이스에 담긴 원소가 있는지 확이하려면 길이를 검사한다., nil과 empty 슬라이스는 모두 길이가 0이므로, 길이 검사로 충분하다.
28. API를 모호하지 않게 설계하려면 nil, empty구분을 하면 안된다.
29. 맵의 메모리 사용량은 증가하기만하고 감소하지 않는다. 메모리 관련 문제가 발생할 수 있으므로 강제로 맵을 새로 생성하거나 포인터를 사용하는 방식을 사용한다.
30. 고 언어에서 타입을 비교할 때 비교 가능한 타입(부울, 숫자, 스트링, 포인터, 채널, 비교 가능한 타입만 있는 구조체)에 대해서 == !=를 사용한다. 나머지는 리플렉션의 오버헤드를 감수할 수 있다면
    reflect.DeepEqual을 사용하고 아니라면 직점 비교 메서드를 구현한다.

### 제어

31. range 루프에서 값 원소는 복제본이다. 따라서 구조체를 변경하려면 인덱스로 접근하거나 for루프를 사용한다. (포인터가 아닐 때)
32. range 연산자에 지정한 표현식은 루프 시작 전 단 한번만 평가된다. 채널 슬라이스 반복문에서 대입이 비효율적으로 처리되는 실수를 피해라
33. 맵을 사용할 때 결과가 항상 일정하게 나오게 하려면 다음과 같은 맵 데이터의 특성을 명심하라, 키 정렬 아님, 추가된 순서 아님, 반복 순서 일정하지 않음, 반복 중 추가된 원소는 랜덤하게 포함됨
34. break나 continue 시 레이블을 사용해 특정 문장을 빠져나올 수 있다.
35. 함수 안에서 루프를 수행하는 로직을 추출하여 Defer를 적절하게 처리한다.

### 스트링

36. 룬은 유니코드의 코드 포인터 개념에 해당하며, 하나의 룬은 여러 바이트로 구성될 수 있다.
37. range 연산자로 스트링을 반복하면 룬을 기준으로 실해오딘다. 이때 시작 인덱스는 해당 룬의 바이트 시퀀스의 시작 인덱스다. 룬 인덱스를 지정하여 접근하려면, []rune으로 변환
38. strings.TrimRight strings.TrimLeft와 strings.TrimSuffix strings.TrimPrefix의 차이를 알아둬라
39. 리스트에 나온 스트링을 이어 붙일때는 빌더를 써야 할당을 막을 수 있다.
40. string변환하지말고 bytes 패키지를 써라
41. 서브스트링 대신 복제를 이용하면 누수를 막을 수 있다. 기존 스트링의 내부 배열을 참조하면서 회수가 안된다.

### 함수와 메서드

42. 리시버 타입으로 값과 포인터 중 결정할 때, 타입이나 변경 필요성, 복제 여부, 오브젝트 크기 등에 따라 결정해야 한다. 잘 모르겠다면 포인터 리시버
43. 기명 결과 매개변수를 사용하면 가독성을 높일수있다. 타입이 같은 결과 매개변수가 여러개일 때 효과적이다. 0으로 초기화되어 편리할 수 있다. 하지만 부작용도 주의하자.
44. 인터페이스를 리턴할 때 nil 포인터가 아닌 nil 값을 리턴하도록 주의한다.호출 측에서 nil이 아닌 값을 받기 때문이다.
45. 함수에서 파일이름 대신 io.Reader를 받도록하여 재사용성을 높이고 테스트 용이성을 높이자.
46. 인수와 리시버가 defer 호출과 동시에 평가되면서, 발생하는 문제를 해결하려면 포인터를 전달하거나 defer 호출문을 클로저로 래핑한다.

### 에러 관리
47. recover()를 호출하여 패닉이 발생한 고루틴을 캡처하는 것은 defer 안에서만 유용하다. 나머지 경우는 nil을 리턴하며 아무런 영향을 받지 않는다. defer는 상위함수에서 패닉이 발생해도 실행되기 때문이다. panic은 프로그래머의 실수처럼 정말 특수한 경우를 알려주는 용도로만 사용, 애플리케이션에 필요한 의존성을 초기화할수없는 경우
48. 에러 래핑은 에러에 문맥추가, 구체화할 때 사용 %w, 하지만 원본 에러에 접근 가능하여 결합이 발생한다. 구현 세부사항에 종속되지 않으려몀ㄴ %v를 쓴다.
49. errorslAs로 래핑된 에러타입을 확인할 수 있다. 재귀적으로 풀어보고 일치하면 참을 반환한다. 제네릭도 있던데...
50. sentinel 에러는 글로벌 변수로 정의한 에러다. 예상한 에러를 표현한다. 예상한 에러는 에러 값으로 예상 못한것은 타입으로 표현한다. 에러 값을 검사할 때 errors.Is를 사용하여 래핑된 에러도 검사한다.
51. 에러는 두번 처리하지 마라, 로그를 남기거나 반환하거나.
52. 에러를 무시할 때는 _로 명시적으로 무시하자.
53. defer에서 발생한 에러를 처리하라. 로그를 남길 수도 있고, 기명 결과 매개변수를 사용해서 호출한 함수에 반환할 수 있다. 단 두가지 에러가 겹친다면 추가적인 처리는 필요하다. 두 에러를 담은 커스텀에러를 반환하거나, 하나는 로그를 남기고 하나는 리턴한다.
