## 실무에서 활용하던 AI 접근법 4단계

기존 실무에서 AI를 활용하며 겪은 주요 장점과 단점, 그리고 이를 시스템적으로 어떻게 제어하고 활용했는지 4가지로 정리했습니다.

### 1: 비정형 데이터 정제

AI의 가장 큰 강점은 비정형 데이터 처리에 있습니다. 사람 역시 비정형 데이터를 잘 해석하지만, 리소스가 많이 들고 단순 반복적인 작업이 되기 쉽습니다. 이러한 업무를 LLM에게 위임했을 때 매우 효과적이었습니다.
> 참고: LLM이 왜 비정형 데이터 구조화에 강한지에 대해서는 추후 별도의 포스팅으로 상세히 다룰 예정입니다.

- 실무 활용 사례:
    - 형태가 일정하지 않은 기획서에서 주요 데이터만 정확히 추출.
    - 사전에 입력 형태를 예측하기 힘든 다양한 코드를 해석하고 리뷰.
- 효과: 비정형 데이터를 정제하여 구조화된 데이터(Structured Output)로 변환해 두면, 이후 프로덕션 코드에서 이를 활용하고 자동화하기가 매우 수월해집니다.

### 2: LLM의 확률적 결과 검증

명확한 단점도 존재합니다. LLM은 확률적으로 텍스트를 생성하기 때문에 비결정론적인(Non-deterministic) 결과를 내며, 이는 할루시네이션 문제와 직결됩니다. 수치와 팩트에 민감한 작업은 AI에게 온전히
위임할 수 없습니다.

- 리스크 사례:
    - 예산 정의: 수치가 포함된 문서를 다룰 때, 결과물에서 수치가 조금이라도 틀어지면 법적/재무적 리스크가 발생합니다.
    - 게임 확률 관리: 기획서의 의도와 실제 코드에 적용된 아이템 획득 확률 수치가 달라진다면 서비스에 치명적인 문제가 생깁니다.

- 해결 방법 (검증):
    - LLM 기반의 크로스체크나 코드 기반의 확실한 안전장치를 마련해야 합니다.
    - 정제 전후 데이터의 총합 비교, 주요 수치의 등장 여부 확인, 테스트 코드 작성 등 스크립트 기반의 결정론적 검증 로직을 추가하는 것이 가장 효과적이었습니다.

### 3: 결정론적 도구 제공 (Deterministic Tool Execution)

LLM의 확률적 특성으로 인해 동일한 입력에도 매번 다른 결과가 나올 수 있으므로, 시스템에서 항상 동일하고 정확한 결과를 보장하기 위한 통제 장치가 필요합니다.

- 해결 방법: LLM에게 직접 연산이나 처리를 맡기는 대신, 스크립트 기반의 도구를 쥐어주고 이를 실행하도록 위임합니다. 이를 통해 AI의 유연성을 활용하면서도 100% 예측 가능한 결과를 얻을 수
  있습니다.
- 실무 활용 사례:
    - 보일러플레이트 코드 생성: 검토 범위를 줄이기 위해, 설정 파일과 스크립트 기반의 코드 생성기를 활용해 항상 예측 가능한 코드를 출력하도록 제어했습니다.
    - 덧붙임: LLM이 생성한 코드를 다시 검증하는 것에 이견이 있을 수 있지만, 저는 검증하는 방식을 선호합니다. 그리고 위 방법은 테스트 코드 작성이 필요없어 리소스가 소모되지 않습니다. 항상 동일한 입력에
      대해 동일한 결과물이 생성되기 때문입니다.

### 4: 최종 결과 생성 (Final Result Generation)

위와 같은 단점과 통제 프로세스에도 불구하고, LLM의 가장 강력한 무기는 단연 '문맥을 이어가는 생성' 능력입니다.

- 실무 활용 사례:
    - 기존 데이터를 분석하여 인사이트를 도출한 결과 보고서 및 요약 작성.
    - 팀 내 학습 자료 및 문서화 자동화.
    - 기존 코드베이스의 컨벤션과 자연스럽게 어울리는 코드 생성.
- 효과: 기존 데이터를 바탕으로 LLM이 새로운 결과물을 생성하게 합니다. 반복 업무 시간을 줄이고, 작업자는 보다 본질적인 문제 해결에 집중할 수 있습니다.

## 시행착오 사례: 범용 크롤링 에이전트 개발기

AI의 비정형 데이터 처리 능력을 맹신하여 적용하다 겪었던 시행착오입니다. '모든 채용 사이트를 알아서 크롤링하는 범용 AI 에이전트'를 만들고자 했던 시도였습니다.

### 배경: AI의 적응력에 대한 기대

채용 공고 홈페이지는 회사마다 DOM 구조와 형태가 제각각입니다. 각 사이트에 맞춘 크롤러를 일일이 짜는 대신, 비정형 데이터를 유연하게 해석하는 AI의 적응력을 활용하려 했습니다. URL만 주면 에이전트가 알아서
데이터를 탐색, 정제하여 구조화된 결과를 내게 만드는 것이 목표였습니다. 이 장점을 극대화하면 최소한의 리소스로 모든 형태의 웹페이지를 커버할 수 있을 것이라 기대했습니다.

### 직면한 문제: 토큰 비용과 웹의 시각적 복잡성

하지만 추론 비용이라는 큰 장벽에 부딪혔습니다. 일반적인 웹페이지는 수많은 시각적 요소와 텍스트 노드를 포함하고 있어, 에이전트가 이를 읽어들이는 데에만 수만 단위의 입력 토큰을 소모했습니다. 게다가 동적
로딩이나 화면 변화를 탐색하는 과정에서 컨텍스트가 누적되며 비용은 기하급수적으로 커졌습니다.

### 문제 재정의와 방향 전환

결론적으로 이는 '잘못된 문제 정의'에서 비롯된 결과였습니다. 수많은 회사의 공고를 한데 모아두는 '채용 포털 사이트'가 이미 존재했기 때문입니다.

- 개별 회사 사이트의 다양한 구조에 대응해야 한다는 전제 자체가 불필요했습니다.
- 개발 리소스는 아꼈을지 몰라도, 실제 운영 비용인 토큰을 심각하게 낭비하고 있었습니다.
- 웹페이지의 구조는 생각보다 단시간에 자주 변경되지 않습니다.

결국 비용이 들지 않는 스크립트 기반의 크롤링으로 방향을 전환했습니다. 채용 포털 단 한 곳만을 타겟팅해 크롤러를 구축했고, 향후 사이트 구조가 변경되어 작동을 멈춘다면 그때만 일회성으로 LLM을 활용해 변경된
구조를 파악하고 코드를 개선하는 방식을 택했습니다.

### 거대 모델을 활용한 소형 모델 디버깅

비록 범용 크롤링 에이전트 구축에는 실패했지만, 이 과정에서 매우 흥미로운 엔지니어링 기법을 하나 발견했습니다. 바로 하이엔드 모델로 실무형 경량화 모델의 문제를 진단하는 방법입니다.

- 현상: 배포하려던 경량화 모델(Gemini-3.1-Flash-Lite)에게 Agent Browser를 쥐어주었을 때, 특정 페이지에 진입하지 못하고 동일한 동작만 반복하는 무한 루프 현상이 로그에서
  발견되었습니다.
- 해결: 경량화 모델이 실패한 행동 궤적(Trajectory)과 동일한 환경 데이터를 하이엔드 모델(GPT-5.4 xHigh)에 컨텍스트로 주입했습니다. 거대 모델은 궤적을 분석하여 정확히 어느 단계에서 판단
  오류가 있었는지 짚어냈습니다.
- 적용: 이 분석 결과를 바탕으로 서비스 중인 소형 에이전트의 프롬프트를 수정하여 정상적인 작동이 가능하도록 디버깅을 완료했습니다.

> 참고: 실제로 그다음 주, 이와 매우 유사한 디버깅 및 설계 전략이 Anthropic(Claude) 공식 블로그를 통해 소개되기도 했습니다.
> 링크: https://claude.com/blog/the-advisor-strategy)

## 이번 스터디의 목표와 해결 과제

이번 스터디의 궁극적인 목표는 "Go 언어의 베스트 프랙티스를 준수하며, 신뢰할 수 있는 추천 시스템을 구축" 하는 것입니다. 지금까지의 활용 경험과 시행착오를 바탕으로, 특히 그동안 깊게 다루지 않았던 AI의
활동을 검증하는 역량을 발전시켜 LLM의 한계를 보완하는 능력을 키우고자 합니다.

### 해결하고자 하는 문제: '작동하는 코드'에서 '좋은 코드'로

기존의 AI 코드 생성기는 '충분히 동작하는' 결과물을 내놓습니다. 하지만 프로젝트 전체의 아키텍처 맥락이나 특정 언어의 철학을 완벽히 반영하기엔 한계가 있습니다. 다른 언어의 패턴(예: 제공자 기반 인터페이스)을
무의식적으로 따라가거나, Go 언어답지 않은 코드를 짤 때가 많습니다.

- 문제점: AI가 생성한 코드가 문법적으로는 완벽하여 포매터나 테스트를 통과하더라도, 성능 저하나 유지보수 문제를 초래하는 안티 패턴을 포함할 위험이 있습니다.
- 해결 방안:  "100 Go Mistakes and How to Avoid Them" 등의 내용을 정리 및 인덱싱하여 RAG 컨텍스트로 활용합니다. 필요할 때마다 Go 베스트 프랙티스를 참조하여 코드를
  발전시키고, 이를 자동으로 검증하는 프로세스를 구축합니다.

### 핵심 구조: 개발과 검증의 에이전트 분리

이번 프로젝트의 핵심은 '코드를 생성하는 AI'와 '이를 검증하는 AI'를 분리하여 상호 견제하게 만드는 것입니다. 동일한 에이전트가 생성과 검증을 모두 수행할 경우, 자신이 만든 코드가 맞다고 편향되게 판단하는 '
확증 편향'이 발생하기 때문입니다.
> Anthropic(Claude)에서 비슷한 방법의 사례를 공개한 적이 있습니다.
> 링크: https://www.anthropic.com/engineering/harness-design-long-running-apps

- 생성: 필요한 Go 패턴의 맥락만 주입받아 초기 코드를 생성합니다.
- 검증 (리뷰어): 설계 의도와 Go 베스트 프랙티스 기준에 맞춰 생성된 코드를 리뷰합니다. PR이나 커밋 시점에 동작하게 하여, 개발자에게 노이즈가 아닌 유의미하고 구조적인 인사이트를 제공합니다.

### 향후 주차별 진행 일정

Go 언어 패턴 학습과 병행하여 다음 일정으로 시스템을 구축할 계획입니다.

- 2주차 (기틀 마련): 공고 추천 에이전트 프로토타입 개발 + 코드 품질 리뷰 기본 시스템 구축
- 3주차 (회고 및 개선): 실제 적용 시의 트러블슈팅 정리 및 시스템 성능 개선
- 4주차 (고도화): 추천 및 검증의 신뢰도 향상
- 5주차 (마무리): 전체 시스템 통합 및 최종 회고 문서 작성
