## 2주차에서 해결하려던 문제

1주차 말미에 제시한 "개발과 검증 에이전트 분리"의 첫 구현으로, **Go 코드 변경분을 받아 한국어 마크다운 리포트를 돌려주는 검토 에이전트**를 만드는 것이 2주차의 목표였습니다. 단순히 "좋은 코드냐 나쁜 코드냐"를 LLM에게 프리폼으로 묻는 방식은 세 가지 한계가 있습니다.

- **비결정성**: 같은 코드에도 매번 다른 부분을 지적합니다.
- **토큰 낭비**: 맥락 없이 매번 프롬프트에 원칙을 다시 밀어 넣어야 합니다.
- **린터와의 중복**: 정적 분석으로 잡을 수 있는 패턴까지 AI에게 맡기는 것은 낭비입니다.

따라서 이번 주차에서는 **"어떤 코드 패턴이 발동 조건"**(트리거)과 **"그 패턴에서 점검할 원칙"**(룰)을 분리해 구조화하고, 린터가 못 잡는 설계/맥락 판단에 한정해 AI에게 위임하는 시스템을 설계했습니다.

## 토큰 소모량을 줄이기 위한 세 가지 축

이번 주차의 많은 설계 결정은 "같은 품질의 리뷰를 더 적은 토큰으로"라는 공통 목표에서 출발했습니다. 서로 다른 레이어에서 각각 아래 세 방식을 병행했습니다.

1. **정적 감지는 린터에 위임.** `golangci-lint`가 이미 완전히 커버하는 룰(Rule 1, 3, 14~17, 22, 38, 49, 52 등)은 스킬 참조에서 빼고, 설정 파일 주석으로 커버리지를 명시했습니다. AI 호출 자체를 없애는 것이 가장 큰 절감입니다.
2. **Skill.md 입·출력량 조정.** 트리거 기반 lazy 로딩으로 **입력** 쪽 레퍼런스 로딩을 발동된 트리거 수만큼으로 제한하고, v1 → v2 포맷 개정으로 **출력** 1건을 4줄 고정으로 압축했습니다. 출력 축소는 이후 턴의 prompt 히스토리를 통째로 줄이는 부수 효과가 큽니다.
3. **부피가 큰 리뷰 작업은 Gemini CLI에 위임.** 코드 리뷰는 한 번에 수십만 토큰을 소비하는 경로라 비싼 Claude 세션에 그대로 태우기엔 부담이 큽니다. Claude 쪽에는 생성·오케스트레이션만 남기고, 실제 리뷰 실행은 Gemini로 분리하는 이원화를 실험했습니다.

이후 섹션들은 각 축의 구체적인 구현과 실측 결과를 다룹니다.

## gosu-review 스킬의 설계 원칙

### 트리거 기반 lazy 로딩

`week2/Skill.md`는 **얇은 라우터**로 동작합니다. 스킬이 활성화되면 파일 전체를 한 번 스캔하며 16개 트리거(함수 시그니처, 에러 처리, defer, 슬라이스 메모리, 제네릭 등) 중 **발동한 것**만 수집하고, 그 트리거에 해당하는 `week2/references/trigger-NN-*.md` 파일만 추가로 읽습니다.

- 16개 룰 본문을 스킬에 모두 박아두면 매 호출마다 수천 토큰이 낭비됩니다.
- 반면 레퍼런스 파일은 평균 50~100줄 분량의 "검사 절차 + 룰 3~7개"로 쪼개져 있어, **발동된 트리거 수만큼만 context 비용을 지불**합니다.
- 4개 이상 트리거가 동시에 발동하고 위치도 11곳 이상이면 **트리거별 서브에이전트에게 분배**하는 모드로 전환합니다 (메인은 조율만).

### 린터와의 역할 분담

Go 생태계에는 이미 강력한 정적 분석기 `golangci-lint`가 존재합니다. AI 리뷰어가 린터가 이미 잡는 영역을 중복 검사하면 가치가 없습니다. `week2/golangci.yaml`은 각 린터에 **룰 매핑 주석**을 붙여 커버리지를 명시합니다.

- `✓` : 해당 룰을 린터가 완전히 커버함 → 스킬에서 제외 (Rule 1, 3, 14~17, 22, 38, 49, 52)
- `~` : 부분 커버 → 스킬과 이중 안전망
- 그 외 : 룰 매핑은 없지만 일반 품질 향상용

즉 **린터 = 패턴 감지, 스킬 = 설계 판단**으로 명확히 선을 그었습니다.

### 검토 리포트의 출력 계약

리포트는 마크다운으로 출력되며, 1건당 다음 4개 필드를 채우도록 강제합니다.

- `파일:라인`
- `룰 번호` 및 `룰 이름`
- `변경 여부` (🔴 변경 라인, 🟡 기존 라인)
- `근거` — 1문장. `제안` — 1줄 요약.

발견 항목이 파일별로 모이고, 같은 파일 안에서는 `🔴 → 🟡` 순, 라인 번호 오름차순으로 정렬됩니다. 트리거 14(패키지 구조)의 발견은 파일 단위가 아니라 **리포트 맨 앞 "프로젝트 구조" 섹션**에 따로 놓입니다.

## 테스트 샘플 설계

### 블라인드 입력 / 정답지 분리

리뷰어의 `recall`과 `precision`을 객관적으로 측정하기 위해 테스트 샘플을 두 갈래로 분리했습니다.

- `week2/testdata/triggerNN_*/fixture.go` — **답안 없음**. 스킬이 힌트 없이도 같은 룰을 잡아내는지 검증하는 주 입력.
- `week2/testgolden/triggerNN_*/fixture.go` — 동일한 코드에 `// Rule N 위반: ...` 주석을 심어둔 정답지. 리뷰어가 놓친 항목을 눈으로 대조할 때 참고.

정답지에 있는데 보고되지 않은 항목은 recall 실패로, 정답지에 없는데 보고된 항목은 precision 실패로 분류합니다.

### 트리거 14 예외 처리

트리거 14(패키지·디렉토리 구조)는 **파일 한 개가 아니라 디렉토리 전체**가 입력입니다. `util/`, `common/`, `pricing/` 같은 의미 없는 범용 이름을 쓰고 있는지, 순환 import가 있는지 등은 프로젝트 전체를 훑어야 판단할 수 있습니다. 따라서 테스트 샘플도 `testdata/trigger14_package_structure/{util,common,pricing}/` 형태로 제공하며, 스킬 워크플로우 상 **이 트리거만큼은 서브에이전트에 위임하지 않고 메인이 직접 처리**하도록 규칙을 뒀습니다.

## 스킬 리팩토링 실험: v1 → v2 포맷 압축

스킬을 작성해 돌려보니 초기 버전(v1)의 리포트가 장황했습니다. `근거`는 2~3문장으로 늘어지고, `제안`은 before/after 코드 블록까지 포함해 50~80줄까지 쉽게 부풀었습니다. 이에 **Skill.md의 출력 포맷 계약을 더 엄격하게 조인 개정(v2)** 을 만들었습니다.

- `근거` : 1~3줄 → **1문장**
- `제안` : 코드 스니펫 포함 권장 → **1줄 요약**, 코드 스니펫은 비자명할 때만 최대 5줄 (변경 후 형태만, before/after 분리 금지)
- 발견 1건을 **4줄 고정** 포맷으로 제약
- 맨 끝에 "포맷 예시 (1건)" 블록을 추가해 앵커 고정

**직관적 예상**: 출력 포맷을 압축했으니 Skill.md 본문도 짧아졌을 것이다.
**실제 결과**: Skill.md 본문은 오히려 **+544 byte / +158~206 토큰**만큼 늘었습니다. "이렇게 써라"는 제약 조건이 구체화되면서 지시문 자체가 길어졌기 때문입니다.

### 측정 파이프라인 구축

초기 시도로 `git show <ref>:week2/Skill.md`를 그대로 gemini stdin에 파이프해 응답의 `tokens.input`을 뽑는 스크립트(`week2/scripts/compare_skill_tokens.sh`)를 만들었지만, 이 방식은 **Skill.md 원문 자체의 raw 토큰 차이만** 보여줄 뿐 실제 리뷰 세션에서의 비용(시스템 프롬프트, 레퍼런스 로딩, 턴 누적 히스토리)을 전혀 반영하지 못합니다. +158~+206 토큰이라는 결과는 참고용으로만 남기고 의사결정에는 쓰지 않았습니다.

실제 검증은 **프로덕션 경로를 그대로 밟는 세션 단위 측정**으로 했습니다.

1. **두 버전을 독립된 스킬로 패키징.** `week2/scripts/skill-variants/gosu-review-v{1,2}/SKILL.md`에 각각 v1/v2 본문을 넣고, `name`과 `description`을 `gosu-review-v1` / `SKILLTESTV1...`, `gosu-review-v2` / `SKILLTESTV2...`로 분리해 트리거 매칭과 식별이 섞이지 않게 했습니다. 측정 대상은 SKILL.md 본문뿐이므로 `references/*.md`는 일부러 복제하지 않았습니다.
2. **세션 간 오염 제거.** 같은 세션에서 v1 → v2를 이어 측정하면 앞 스킬의 metadata와 대화 히스토리가 그대로 누적되어 diff가 섞입니다. 따라서 각 스킬을 추가(install + enable)한 뒤에는 **반드시 gemini 세션을 새로 열어** 트리거 문장 1회만 보내고 `/stats`를 찍는 방식으로 진행했습니다. `uninstall` 단계는 따로 두지 않았고, 측정 사이의 격리는 "새 세션을 여는 것"만으로 확보했습니다.
3. **모델·입력 통제.** 첫 시도에서 v1은 `gemini-3.1-pro-preview`, v2는 `gemini-3-flash-preview`로 자동 라우팅돼 단가·토큰화가 달라 비교가 무의미했습니다. 이를 발견한 뒤 v2를 같은 `gemini-3.1-pro-preview`로 재측정했고, 트리거 문장도 v1/v2에서 글자 수를 맞춘 한 줄로 고정했습니다.
4. **`/stats` 스냅샷 비교.** Gemini 인터랙티브 세션의 `/stats`는 해당 세션 한정 누적을 출력합니다. v1/v2 각 세션의 `main` 역할 `prompt / cached / input / output` 네 축을 그대로 기록해 표로 정리했습니다.

### 결과와 해석

동일 모델 기준 **세션 누적 토큰**:

| 항목 | v1 | v2 | diff |
|---|---|---|---|
| main 요청 수 | 13 | 11 | −2 |
| prompt 누적 | 315,781 | 216,404 | **−99,377 (−31.5%)** |
| cached 누적 | 217,314 | 135,675 | −81,639 |
| output 누적 | 8,430 | 6,481 | **−1,949 (−23.1%)** |
| 1요청 평균 prompt | 24,291 | 19,673 | −4,618/req |
| 1요청 평균 non-cached 입력 | 7,574 | 7,339 | −235/req |

Skill.md 본문 자체는 v2가 +544 byte 더 크지만, **출력 포맷을 압축한 덕에 매 턴의 응답 길이가 짧아졌고**, 그 짧은 응답이 다음 턴의 히스토리로 누적되지 않아 **세션 전체 토큰이 −31.5% 감소**했습니다. 지시문 몇 줄을 더 넣고 빼는 것보다 모델의 출력 길이를 제약하는 쪽이 누적 비용에 훨씬 크게 기여한다는 점을 실측으로 확인한 것이 이번 실험의 의의입니다. 다만 절대량 기준으로는 여전히 입력(prompt + cached)이 출력의 수십 배이므로, 다음 단계에서는 **레퍼런스 로딩 범위를 더 좁히거나 컨텍스트 재사용을 최적화해서 입력 비용 자체를 줄이는 작업**도 함께 진행되어야 합니다.

### 리뷰 품질의 교차 검증

토큰을 아껴도 검출력이 떨어지면 의미가 없습니다. 3개 트리거(1, 3, 5)에 대해 v1과 v2의 리포트를 직접 대조했습니다.

| 트리거 | 커버리지 | 라인 번호 정확도 | 리포트 길이 |
|---|---|---|---|
| 01 함수 시그니처 | 양쪽 7/7 | v1 −1 균일 / v2 −4~−9 편차 | v1 장문 / v2 간결 |
| 03 defer | v1 4/4, v2 3/4 + Rule 53 신규 1건 | v1 −1 / v2 ±1~2 | v1 장문 / v2 간결 |
| 05 슬라이스·문자열 | 양쪽 7/7 | v1 −1~−3 / v2 **±1 이내 (v2 승)** | v1 장문 / v2 간결 |

**해석**: v2는 포맷 간결화로 토큰을 크게 절감

## 이번 주차에서 얻은 교훈

- **스킬 본문을 줄이는 것보다, 스킬이 유도하는 출력 분포를 좁히는 편이 누적 비용 관점에서 훨씬 효과적이다.** 짧은 출력은 이후 턴의 prompt 히스토리를 통째로 축소시킨다.
- **린터와 AI 리뷰어는 역할이 다르다.** 정적 감지 가능한 패턴을 AI에 맡기는 것은 낭비이므로, 커버리지를 설정 파일에 명시적으로 문서화하는 것이 장기 유지보수에 유리하다.
- **테스트 샘플 분리(blind vs golden) 가 평가를 가능하게 한다.** 정답지를 코드 주석에 심어두면 나중에 recall/precision을 수작업으로라도 빠르게 점검할 수 있다.
- **토큰 측정은 반드시 실제 세션 경로(/stats)에서 재확인해야 한다.** SKILL.md 스크립트 측정만으로는 "캐시·히스토리 누적" 효과를 놓칠 수 있다.
- **Opus 기준 출력은 입력보다 약 5배 비싸다.** 하지만 이번 세션처럼 입력(prompt + cached) 총량이 출력을 수십 배 넘어서는 구조에서는, 단가가 싼 입력 쪽이라도 총량을 줄이는 것이 전체 비용에 더 크게 기여한다. v2에서 prompt를 −99,377 줄인 것의 금액 효과가 output을 −1,949 줄인 것과 비교해 결코 작지 않은 이유다.
- **Gemini에 리뷰 작업을 위임해 Claude 쪽 토큰을 줄이는 구성도 유효한 전략이다.** 이번 측정에서 확인했듯 코드 리뷰 경로의 토큰 소비량은 Gemini 쪽이 수십만 단위로 훨씬 크므로, 비싼 Claude 컨텍스트는 생성·오케스트레이션에 남기고 부피가 큰 정적 리뷰는 Gemini CLI로 넘기는 이원화가 합리적이다.
