## 2주차에서 해결하려던 문제

1주차 말미에 제시한 "개발과 검증 에이전트 분리"의 첫 구현으로, **Go 코드 변경분을 받아 한국어 마크다운 리포트를 돌려주는 검토 에이전트**를 만드는 것이 2주차의 목표였습니다. 단순히 "좋은 코드냐 나쁜 코드냐"를 LLM에게 프리폼으로 묻는 방식은 세 가지 한계가 있습니다.

- **비결정성**: 동일 입력에도 매번 다른 포인트를 집음.
- **토큰 낭비**: 맥락 없이 매번 프롬프트에 원칙을 재주입.
- **린터와의 중복**: 이미 정적 감지가 가능한 패턴을 AI에게 맡기는 것은 비효율.

따라서 이번 주차에서는 **"어떤 코드 패턴이 발동 조건"**(트리거)과 **"그 패턴에서 점검할 원칙"**(룰)을 분리해 구조화하고, 린터가 못 잡는 설계/맥락 판단에 한정해 AI에게 위임하는 시스템을 설계했습니다.

## gosu-review 스킬의 설계 원칙

### 트리거 기반 lazy 로딩

`week2/Skill.md`는 **얇은 라우터**로 동작합니다. 스킬이 활성화되면 파일 전체를 한 번 스캔하며 16개 트리거(함수 시그니처, 에러 처리, defer, 슬라이스 메모리, 제네릭 등) 중 **발동한 것**만 수집하고, 그 트리거에 해당하는 `week2/references/trigger-NN-*.md` 파일만 추가로 읽습니다.

- 16개 룰 본문을 스킬에 모두 박아두면 매 호출마다 수천 토큰이 낭비됩니다.
- 반면 레퍼런스 파일은 평균 50~100줄 분량의 "검사 절차 + 룰 3~7개"로 쪼개져 있어, **발동된 트리거 수만큼만 context 비용을 지불**합니다.
- 4개 이상 트리거가 동시에 발동하고 위치도 11곳 이상이면 **트리거별 서브에이전트에게 분배**하는 모드로 전환합니다 (메인은 조율만).

### 린터와의 역할 분담

Go 생태계에는 이미 강력한 정적 분석기 `golangci-lint`가 존재합니다. AI 리뷰어가 린터가 이미 잡는 영역을 중복 검사하면 가치가 없습니다. `week2/golangci.yaml`은 각 린터에 **룰 매핑 주석**을 붙여 커버리지를 명시합니다.

- `✓` : 해당 룰을 린터가 완전히 커버함 → 스킬에서 제외 (Rule 1, 3, 14~17, 22, 38, 49, 52)
- `~` : 부분 커버 → 스킬과 이중 안전망
- 그 외 : 룰 매핑은 없지만 일반 품질 향상용

즉 **린터 = 패턴 감지, 스킬 = 설계 판단**으로 명확히 선을 그었습니다.

### 검토 리포트의 출력 계약

리포트는 마크다운으로 출력되며, 1건당 다음 4개 필드를 채우도록 강제합니다.

- `파일:라인`
- `룰 번호` 및 `룰 이름`
- `변경 여부` (🔴 변경 라인, 🟡 기존 라인)
- `근거` — 1문장. `제안` — 1줄 요약.

발견 항목이 파일별로 모이고, 같은 파일 안에서는 `🔴 → 🟡` 순, 라인 번호 오름차순으로 정렬됩니다. 트리거 14(패키지 구조)의 발견은 파일 단위가 아니라 **리포트 맨 앞 "프로젝트 구조" 섹션**에 따로 놓입니다.

## 테스트 샘플 설계

### 블라인드 입력 / 정답지 분리

리뷰어의 `recall`과 `precision`을 객관적으로 측정하기 위해 테스트 샘플을 두 갈래로 분리했습니다.

- `week2/testdata/triggerNN_*/fixture.go` — **답안 없음**. 스킬이 힌트 없이도 같은 룰을 잡아내는지 검증하는 주 입력.
- `week2/testgolden/triggerNN_*/fixture.go` — 동일한 코드에 `// Rule N 위반: ...` 주석을 심어둔 정답지. 리뷰어가 놓친 항목을 눈으로 대조할 때 참고.

누락은 recall 결함, 심지 않은 위반을 보고하면 precision 결함으로 분류합니다.

### 트리거 14 예외 처리

트리거 14(패키지·디렉토리 구조)는 **파일 한 개가 아니라 디렉토리 전체**가 입력입니다. `util/`, `common/`, `pricing/` 같은 의미 없는 범용 이름을 사용하는지, 순환 import가 있는지 등 전역 시야가 필요합니다. 따라서 테스트 샘플도 `testdata/trigger14_package_structure/{util,common,pricing}/` 형태로 제공하며, 스킬 워크플로우 상 **이 트리거만큼은 서브에이전트에 위임하지 않고 메인이 직접 처리**하도록 규칙을 뒀습니다.

## 스킬 리팩토링 실험: v1 → v2 포맷 압축

스킬을 작성해 돌려보니 초기 버전(v1)의 리포트가 장황했습니다. `근거`는 2~3문장으로 흐르고, `제안`은 before/after 코드 블록까지 포함해 쉽게 50~80줄로 부풀었습니다. 이에 **Skill.md의 출력 포맷 계약을 조이는 개정(v2)** 을 가했습니다.

- `근거` : 1~3줄 → **1문장**
- `제안` : 코드 스니펫 포함 권장 → **1줄 요약**, 코드 스니펫은 비자명할 때만 최대 5줄 (변경 후 형태만, before/after 분리 금지)
- 발견 1건을 **4줄 고정** 포맷으로 제약
- 맨 끝에 "포맷 예시 (1건)" 블록을 추가해 앵커 고정

**직관적 예상**: 출력 포맷을 압축했으니 Skill.md 본문도 짧아졌을 것이다.
**실제 결과**: Skill.md 본문은 오히려 **+544 byte / +158~206 토큰 증가**. 지시문이 구체화되며 길어졌기 때문.

### 측정 파이프라인 구축

두 가지 방식으로 토큰 비용을 측정했습니다.

1. **stdin 파이프 스크립트** (`week2/scripts/compare_skill_tokens.sh`)
   `git show <ref>:week2/Skill.md | gemini --approval-mode plan -p "ok" -o json`으로 각 버전의 본문을 gemini에 넘기고, JSON 응답에서 `stats.models[].roles.main.tokens.input`을 `jq`로 뽑아 diff.
   - stdin 내용이 지시문으로 해석되어 `run_shell_command` 도구 호출을 무한 반복하는 문제를 `--approval-mode plan`(읽기 전용)로 해결.
   - v1 vs v2 본문 자체의 raw 토큰 차이: **+158~+206 토큰** (런간 ±30 노이즈).

2. **실제 스킬 설치 + /stats 세션 측정**
   `week2/scripts/skill-variants/gosu-review-v{1,2}/SKILL.md`로 v1/v2를 각각 별도 스킬로 패키징하고, 독립 세션에서 `gemini skills install → 트리거 → /stats → /quit → uninstall` 절차로 세션 누적을 비교.
   - 같은 모델(`gemini-3.1-pro-preview`)로 통제 후 비교.

### 결과와 해석

동일 모델 기준 **세션 누적 토큰**:

| 항목 | v1 | v2 | diff |
|---|---|---|---|
| main 요청 수 | 13 | 11 | −2 |
| prompt 누적 | 315,781 | 216,404 | **−99,377 (−31.5%)** |
| cached 누적 | 217,314 | 135,675 | −81,639 |
| output 누적 | 8,430 | 6,481 | **−1,949 (−23.1%)** |
| 1요청 평균 prompt | 24,291 | 19,673 | −4,618/req |
| 1요청 평균 non-cached 입력 | 7,574 | 7,339 | −235/req |

**결론의 반전**: Skill.md 본문 자체는 v2가 더 크다. 그러나 **출력 포맷 압축이 매 턴의 응답을 축소**시키고, 그 짧은 응답이 다음 턴의 히스토리로 누적되지 않아 **세션 전체 토큰 사용이 −31.5% 감소**했습니다. "스킬 본문 크기 ≠ 실제 토큰 비용"이라는 비자명한 결론을 실측으로 얻은 것이 이번 실험의 핵심 성과입니다.

### 리뷰 품질의 교차 검증

토큰을 아껴도 검출력이 떨어지면 의미가 없습니다. 3개 트리거(1, 3, 5)에 대해 v1과 v2의 리포트를 직접 대조했습니다.

| 트리거 | 커버리지 | 라인 번호 정확도 | 포맷 |
|---|---|---|---|
| 01 함수 시그니처 | 양쪽 7/7 | v1 −1 균일 / v2 −4~−9 편차 | v1 장문 / v2 간결 |
| 03 defer | v1 4/4, v2 3/4 + Rule 53 신규 1건 | v1 −1 / v2 ±1~2 | 동일 경향 |
| 05 슬라이스·문자열 | 양쪽 7/7 | v1 −1~−3 / v2 **±1 이내 (v2 승)** | 동일 경향 |

**해석**: v2는 포맷 간결화로 토큰을 크게 절감했지만, 트리거 01처럼 라인 번호 트래킹이 필요한 경우에 정확도가 벌어지는 트레이드오프가 관찰되었습니다. 이는 Skill.md 개정 효과와 **모델 선택의 영향이 섞여** 나타난 현상일 수 있으므로, 다음 주차에 모델을 더 엄격히 고정한 재측정이 필요합니다.

## 이번 주차에서 얻은 교훈

- **스킬 본문을 줄이는 것보다, 스킬이 유도하는 출력 분포를 좁히는 편이 누적 비용 관점에서 훨씬 효과적이다.** 짧은 출력은 이후 턴의 prompt 히스토리를 통째로 축소시킨다.
- **린터와 AI 리뷰어는 역할이 다르다.** 정적 감지 가능한 패턴을 AI에 맡기는 것은 낭비이므로, 커버리지를 설정 파일에 명시적으로 문서화하는 것이 장기 유지보수에 유리하다.
- **테스트 샘플 분리(blind vs golden) 가 평가를 가능하게 한다.** 정답지를 코드 주석에 심어두면 나중에 recall/precision을 수작업으로라도 빠르게 점검할 수 있다.
- **토큰 측정은 반드시 실제 세션 경로(/stats)에서 재확인해야 한다.** SKILL.md 스크립트 측정만으로는 "캐시·히스토리 누적" 효과를 놓칠 수 있다.
- **Opus 기준 출력은 입력보다 약 5배 비싸다.** 다만 이번 세션의 실제 사용량은 입력(prompt+cached)이 출력을 압도하는 구조이므로, 단가가 싼 쪽이라도 입력 총량을 더 줄일 여지를 찾는 것이 비용 효율 관점에서 우선이다. v2의 −99,377 prompt 토큰 감소가 −1,949 output 감소보다 금액 기여가 작지 않은 이유다.
- **Gemini 쪽에 리뷰 작업을 위임해 Claude 쪽 토큰을 줄이는 구성도 유효한 전략이다.** 이번 측정에서 확인됐듯 코드 리뷰 경로의 토큰 소비량은 Gemini 쪽이 훨씬 크므로(수십만 단위), 비싼 Claude context는 생성·오케스트레이션에, 부피가 큰 정적 리뷰는 Gemini CLI 리소스에 배치하는 이원화가 합리적이다.

## 다음 주차 진행 계획
