# Trigger 02: 에러 처리 (`if err != nil`)

> 이 파일은 gosu-review SKILL의 트리거 02 레퍼런스다.
> 메인 SKILL.md의 워크플로우 3단계에서 이 트리거가 발동하면 읽는다.

## 발동 조건

- `if err != nil { ... }` 블록 (변수명이 `err`가 아니어도 동일 패턴 포함)
- 타입 어설션 `err.(*MyError)`, `err.(MyError)` (Rule 49)
- `==` 또는 `!=`로 에러 값을 직접 비교 (Rule 49 — `errors.Is` 대체 후보)

## 검사 절차

1. 블록 내부의 처리 흐름 확인 — 로깅과 반환이 동시에 있는가? (Rule 51)
2. `fmt.Errorf` 또는 에러 래핑이 있으면 동사(`%w` vs `%v`) 검토 (Rule 48)
3. 에러 타입을 분기하는 코드에서 타입 어설션을 쓰는지, 값 비교를 `==`로 하는지 확인 (Rule 49)
4. 같은 위치에 트리거 4(중첩)가 함께 발동하면 같이 본다

## 룰

### Rule 51 — 에러 이중 처리
- 핵심 질문: 같은 에러를 로깅도 하고 반환도 하는가?
- 문제: 호출 스택 위에서 또 처리하면 같은 에러가 여러 번 로깅됨
- 원칙: **둘 중 하나만** — 처리 책임이 있는 계층에서만 로깅, 그 외는 래핑해서 반환
- 예외: 계층 경계에서 메트릭/추적 목적으로 명시적 기록

### Rule 48 — 에러 래핑 전략 (`%w` vs `%v`)
- 핵심 질문: `fmt.Errorf`에서 동사 선택이 의도와 맞는가?
- `%w`: 원본 에러를 결합 → `errors.Is`/`errors.As` 가능, 그러나 호출 측이 내부 에러 타입에 결합됨
- `%v`: 단순 문맥 추가 → 결합 차단, 메시지만 보존
- 가이드: 호출자가 분기해야 하는 경계 에러는 `%w`, 내부 구현 디테일이면 `%v`

### Rule 49 — `errors.Is` / `errors.As` 활용
- 핵심 질문: 래핑된 에러 체인을 풀어 검사해야 하는 자리에서 `errors.Is`/`errors.As`를 쓰고 있는가?
- 안티패턴:
  ```go
  if err == io.EOF { ... }          // 래핑되면 실패
  if e, ok := err.(*MyError); ok {  // 래핑되면 실패
      ...
  }
  ```
- 권장:
  ```go
  if errors.Is(err, io.EOF) { ... }          // 값 비교
  var e *MyError
  if errors.As(err, &e) { ... }              // 타입 추출
  ```
- 가이드:
    - 특정 **값**과 비교 → `errors.Is` (센티널 에러 — Rule 50 참고)
    - 특정 **타입**으로 변환 → `errors.As` (재귀적으로 풀어서 일치 시 대입)
- 예외: 래핑이 절대 발생하지 않는 폐쇄된 경계 (자기 패키지 내부 즉시 반환 등)
