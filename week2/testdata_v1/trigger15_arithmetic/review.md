## fixture.go

### 🟡 기존 라인의 문제 (참고)

#### L5 `return int32(n)`
**[Rule 18] 정수 오버/언더플로**
- 근거: 외부 입력 문자열을 파싱한 `int` 타입 변수 `n`을 명시적으로 더 작은 `int32`로 형변환하고 있습니다. `n`이 `int32`의 범위를 넘어설 경우 오버플로가 발생하여 조용히 값이 잘립니다.
- 제안: 형변환 전에 `n`이 `math.MaxInt32`와 `math.MinInt32` 범위 내에 있는지 검증하세요.

#### L9 `return price == expected`
**[Rule 19] 부동소수점 직접 비교 지양**
- 근거: 부동소수점(`float64`) 변수를 `==` 연산자로 직접 비교하고 있습니다. 부동소수점 연산은 미세한 오차가 발생할 수 있어 직접 비교 시 의도치 않게 `false`가 될 수 있습니다.
- 제안: 엡실론(epsilon) 범위 비교를 사용하세요.
  ```go
  // before
  	return price == expected
  // after
  	const eps = 1e-9
  	return math.Abs(price-expected) < eps
  ```

#### L15 `s = s + t`
**[Rule 20] 부동소수점 연산 순서**
- 근거: 큰 값(`bigVal`)에 작은 부동소수점 값(`tinies`)들을 계속 더하면, `float64` 정밀도 한계로 인해 작은 값들이 무시되는 정밀도 손실(자릿수 손실)이 발생할 수 있습니다.
- 제안: 작은 값들끼리 먼저 더한 후 나중에 큰 값과 더하는 방식(Kahan 합산 알고리즘 등)을 검토하세요.
