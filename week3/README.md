# Week 3: 데이터 기반 정비와 LLM을 쓰지 않기로 한 결정

## 개요

이번 주차는 추천 시스템의 **데이터 기반을 다지는 작업**이 대부분을 차지했습니다. 그 과정에서 LLM을 끼워 넣으려던 **회사명 정규화 시도를 폐기한 의사결정**이 이번 주차의 가장 의미 있는 산출물이 되었습니다. 폐기 사유를 정리하다 보니 1주차에 폐기했던 범용 크롤러 시도와도 같은 형태의 판단을 거치고 있었음을 알게 되어, 두 사례를 묶어 **"결정론적 문제 모임의 유한성 기준"** 이라는 LLM 적용 여부 판단 규칙으로 일반화했습니다. 보조로 skipjd 시스템 구축 작업, 2주차 산출물(`gosu-review`)을 실제 코드에 적용하며 발견한 데이터 컷오프 한계 사례를 덧붙입니다.

## 2주차 계획과 실제 3주차 진행의 격차

2주차 README 말미에 적은 3주차 계획은 *"추천 에이전트를 Go 코드에 실제 적용하며 개선하고, 채용 공고 추천을 위한 데이터 수집·매칭 기준·추천 파이프라인의 초안을 잡는다"* 였습니다. 그러나 실제로 이번 주차에 한 일은 추천 파이프라인의 *위쪽*이 아니라 *아래쪽*, 즉 **데이터 기반을 정비하는 시스템 구축**이었습니다.

- **계획**: 추천 에이전트 적용 + 채용 공고 추천 작업 착수.
- **실제**: 채용 공고 하나에 여러 직무 코드를 붙일 수 있도록 스키마와 스크래퍼를 확장, gamejob 공고 본문과 이미지 수집(iframe 직접 조회 + 이미지 공고는 PaddleOCR 워커 위임), 회사명 정규화, 사용자 선호도·경력 모델 추가.
- **사유**: 추천 파이프라인을 만들기 전에 입력 데이터의 신뢰성과 매칭 가능한 형태(시스템이 미리 인지하는 닫힌 공간 안의 정형 데이터)가 먼저 필요했습니다. 특히 회사명 코드화에 LLM을 끼워 넣으려다 *"이 문제는 LLM이 풀어야 할 문제가 아니다"* 라는 결론에 이르렀고, 이 판단이 다음 섹션의 메인 콘텐츠가 됩니다.

계획에서 벗어난 폭 자체보다 **왜 벗어났는지에서 얻은 학습**이 이번 주차의 결과물입니다.

## AI 적용을 폐기한 사례: 사용자 입력 회사명 정규화

### 시도했던 파이프라인

사용자가 입력한 회사명(예: "스마일게이트", "휘찬게임즈")을 사내 회사 ID로 매핑하기 위해 LLM을 끼워 넣은 3단계 파이프라인을 설계했습니다.

```
[ 사용자 회사명 입력 ] ── (예: "스마일게이트", "휘찬게임즈")
        │
        ▼
[ 1단계: 사전 및 캐시 조회 ] ── (동의어 사전 및 Redis 캐시 확인)
        │
        ├─ (매핑 데이터 존재) ──► [ 처리 완료 ] ─► 사용자 DB에 매핑된 회사 ID 저장
        │
        ▼ (매핑 데이터 없음)
[ 2단계: 후보군 추출 ] ── (알고리즘 기반 유사도 검색을 통한 Top-N 후보 도출)
        │
        ▼
[ 3단계: LLM 기반 정밀 매칭 ] ── (입력 원문과 Top-N 후보군 교차 검증)
        │
        ├─ (후보 내 정답 존재) ──► [ 동의어 사전 추가 ] ─► 사용자 DB에 회사 ID 저장
        │
        └─ (후보 내 정답 없음) ──► [ Miss 로그 기록 ] ─► [ 예외 처리 (에러 반환) ]
```

설계 자체는 그럴듯했지만, 이 파이프라인을 실제로 운영해야 하는 이유가 빈약했습니다. 폐기로 이어진 사고 과정을 정리해 둡니다.

### 폐기로 이어진 판단

비정형 데이터의 무질서를 백엔드에서 복구하려고 LLM 비용을 지불하는 대신, 입력 폼에 제약을 걸어 무질서의 발생 자체를 막는 쪽이 더 단순했습니다. 사용자에게 자유 입력을 허용하지 않고, 크롤러가 이미 모아 둔 회사 목록에서 골라 선택하게 하면, 뒷단의 LLM 매칭은 애초에 일어날 일이 없어집니다.

이런 결정이 가능했던 이유는 데이터 생산자(채용 공고 크롤링·등록 측)가 이미 정규화된 회사 목록을 굳건히 갖추고 있기 때문이었습니다. 소비자(사용자 입력 측)는 생산자가 만들어 둔 그 목록을 단순 참조하면 끝이고, 양쪽에 정제 파이프라인을 중복으로 둘 이유가 없었습니다. 결과적으로 시스템은 언제든 새로운 값이 들어올 수 있다는 열린 공간의 불확실성을 다루지 않고, 이미 인지한 데이터 풀 안에서만 선택하는 닫힌 공간의 문제로 재정의됩니다. 해결해야 할 문제 자체가 소멸한 셈입니다. 닫힌 공간으로 만들 수 있다면 LLM 단계 자체를 제거할 수 있다는 것이 이번 사례의 함의입니다.

## 언제 LLM을 쓸 것인가: 결정론적 문제 모임의 유한성 기준

회사명 정규화 폐기 결정을 내리고 보니, 이번 주차의 의사결정과 1주차의 범용 크롤링 에이전트 시행착오가 모두 같은 형태의 판단을 거치고 있었다는 것을 알게 됐습니다. 이번 섹션에서는 두 사례를 묶어 LLM을 쓸지 말지 결정할 때 먼저 던져 봐야 할 질문을 정리합니다.

### 1주차 회고와 연결

1주차에서 폐기했던 *"URL만 주면 알아서 크롤링하는 범용 AI 에이전트"* 시도의 본질은, **각 채용 사이트의 DOM 구조가 제각각이다**라는 다양성에 LLM의 적응력으로 대응하려 한 것이었습니다. 당시 회고는 "토큰 비용이 너무 컸다"에 머물렀습니다. 3주차의 회사명 정규화 폐기를 거치며 한 단계 더 들어간 회고가 가능해졌습니다.

- 결정론적 문제(특정 DOM에서 특정 필드를 추출하는 것)에 코드가 더 적합하다는 사실은 *원래부터* 알고 있었습니다.
- LLM을 고려한 진짜 동기는 "결정론적 문제 *여러 개*를 동시에 처리해야 한다"는 점이었습니다. 즉, 다뤄야 할 대상은 결정론적 문제 *그 자체* 가 아니라 결정론적 문제 *여러 종류의 묶음* 이었습니다.
- 채용 공고를 모아둔 포털 사이트의 존재가 확인되자, 다뤄야 할 결정론적 문제가 **코드로 감당할 만한 몇 가지**(포털 1곳의 페이지 구조 몇 가지)로 줄었습니다. 이 순간 LLM의 효용이 사라졌습니다.

### 채용 포털 존재의 의미

"여러 사이트를 본다"가 "한 포털을 본다"로 줄어든 것은 단순한 *대상 축소*가 아닙니다. **풀어야 할 문제 종류가 끝없이 늘어나는 상태에서 코드로 감당할 수 있는 몇 가지로 좁혀진 사건**이며, 이것이 LLM 적용 여부의 결정적 변수였습니다.

### 다른 사례: 부서별 제출 데이터 형식의 다양성

같은 판단 기준이 회사 내부의 데이터 수집 시나리오에도 그대로 적용됩니다.

- *단일 부서가 형식을 자주 바꾼다*: 시점마다 형식이 달라지므로 다뤄야 할 결정론적 문제가 사실상 끝없이 늘어남 → LLM이 유효합니다.
- *각 부서마다 형식은 다르지만 부서별로는 고정*: 부서가 몇 개든 그 수만큼만 결정론적 문제가 생기고, 코드로 감당할 수 있는 규모라면 코드로 대응하는 것이 충분합니다.

핵심은 "형식이 여러 개냐"가 아니라 **"그 여러 개가 우리가 코드로 감당할 수 있는 규모로 추려지는가"** 입니다. 종류가 유한하더라도 그 수가 너무 많으면 결국 LLM에 맡기는 게 현실적입니다.

### 판단 규칙 정리

이번 주차 결론을 다음의 룰로 정리합니다.

- **종류가 코드로 감당할 만큼 적게 추려진다 → 코드.** 각 문제마다 결정론적 코드를 작성하면 됩니다. 비용·재현성·검증 가능성 모두 코드 우위입니다.
- **종류가 사실상 끝이 없거나, 유한하더라도 코드로 감당하기엔 너무 많다 → 두 가지 선택지가 있습니다.**
    1. *비결정적 문제로 간주하고 LLM에 위임*. 적응력을 사용합니다.
    2. *문제 종류를 코드로 감당할 만큼 좁힌 뒤 코드로 처리*. 회사명 정규화 사례에서 "입력을 닫힌 공간으로 묶기"가, 1주차 크롤링 사례에서 "채용 포털 단일 타겟팅"이 정확히 이 좁히기에 해당합니다.
- **종류를 좁힐 수 있는지 먼저 묻습니다.** LLM 적용 여부보다 앞단의 질문이며, 좁히기에 성공하면 LLM 자체가 필요 없어집니다.

이 룰은 2섹션에서 회사명 정규화 시도를 폐기한 판단이 *왜* 작동했는지에 대한 한 단계 위에서의 설명이기도 합니다. 입력 폼에 제약을 걸어 닫힌 공간으로 만들고, 생산자가 만든 회사 목록을 소비자가 그대로 참조하게 한 것은 모두 **다뤄야 할 결정론적 문제의 종류를 코드로 감당할 만큼 좁히는 구체적인 수단**이었습니다.

## skipjd 시스템 구축 작업 정리

이번 주차에 `skipjd` 저장소(추천 시스템 본체)에서 진행한 시스템 구축 작업을 네 갈래로 정리합니다. 위 섹션에서 정리한 판단 규칙을 *왜 이 시스템을 이렇게 만들었는가* 와 묶어 읽으면 좋습니다.

1. **직무 코드 다중 지원**
   모델·에이전트 출력 스키마에 직무 코드 필드를 추가하고, 채용 공고당 여러 개의 직무 코드를 수집·정규화하도록 스크래퍼를 확장했습니다. 다중 직무 코드를 크롤러 파이프라인과 데이터베이스 트랜잭션에 통합했고, 미사용 에이전트 코드를 정리하면서 출력 스키마를 크롤러 패키지로 이동시켰습니다. 직무 코드는 추천 파이프라인의 결정론적 매칭 키 중 하나가 됩니다.

2. **크롤러·스크래핑 개선**
   크롤러 구성에 Functional Options 패턴을 도입해 설정을 유연하게 만들고, gamejob의 `GI_Read_Comt_Ifrm` iframe을 직접 조회해 채용 공고 본문과 이미지를 수집하도록 추가했습니다. 이미지로만 된 공고는 Python PaddleOCR 워커로 위임해 처리합니다. 마지막으로 크롤러 내부의 변수 선언을 정리하고 중복 할당을 제거했습니다.

3. **데이터 정규화**
   법인 식별자(`주식회사`, `(주)` 등)를 제거하는 회사명 정규화 함수를 추가하고, 기존 데이터를 위한 회사명 정규화 마이그레이션 도구를 구현했습니다. 생산자(크롤러) 단계에서 미리 표면적인 무질서를 제거해, 소비자(매칭 로직) 측이 단순 비교만 해도 되도록 만든 부분입니다.

4. **사용자 선호도·경력 모델 추가**
   `seed-preferences` 명령어와 사용자 선호도 데이터 모델을 추가하고, 사용자의 경력 연차를 저장하는 1:1 관계의 `UserCareer` 모델과 연차 설정 플래그를 도입했습니다. 추천 파이프라인이 사용자 측 입력을 닫힌 공간 안의 정형 데이터로 받아들이도록 하기 위한 사전 작업입니다.

## gosu-review 실전 적용 회고

2주차에 만든 `gosu-review` 스킬을 3주차의 실제 Go 코드(`skipjd/internal/crawler` 등)에 적용하며, 잘 잡힌 항목과 부정확하게 잡힌 항목을 모두 수집했습니다.

### 잘 잡은 사례

- **`internal/crawler/crawler.go`** — Rule 7 / 11 / 48
  패키지 진입점인 `NewCrawler`와 `Run`이 인터페이스 대신 구체 타입(`*repository.CrawlerRepository`)에 의존해 모킹이 어렵고, 정의해 둔 Functional Option 패턴을 활용하지 않은 채 `out io.Writer`를 직접 받고 있다는 점을 잡았습니다. 또 `Crawler.Run`의 여러 지점에서 호출자가 분기할 필요가 없는 내부 I/O 에러(`fmt.Fprintf` 실패 등)를 `%w`로 래핑해 내부 구현 디테일을 외부로 누출하고 있는데, 단순 문맥 추가가 목적이라면 `%v`로 충분합니다.
- **`internal/crawler/output.go`** — Rule 48
  `Parse`·`validatePosting`도 같은 방식으로 호출자가 분기할 필요가 없는 내부 파싱·검증 에러(`validatePosting`, `buildSourceKey`, `url.Parse` 등)까지 `%w`로 래핑하고 있어, `%v`로 낮춰도 되는 자리에 `%w`가 들어가 있다는 점을 잡았습니다.
- **`internal/gamejob/detail_scraper.go`** — Rule 46 / 2
  `defer resp.Body.Close()`가 등록 시점에 리시버(`resp.Body`)를 즉시 평가한다는 점(의도적으로 미루려면 클로저로 감싸야 한다는 의미), 그리고 `ExtractDetail`의 `switch case` 안에 `if`가 연속 중첩되어 들여쓰기가 3단계 이상 깊어진 부분을 가드 절로 좌측 정렬할 여지가 있다는 점을 함께 잡았습니다.

이 사례들은 모두 린터가 잡지 못하는 **설계·문맥 판단** 영역이라 2주차에서 의도한 *"린터 = 패턴 감지, 스킬 = 설계 판단"* 의 역할 분담이 실제로 작동했음을 보여줍니다.

### 데이터 컷오프 한계: 잘못된 옛 Go 정보 사례

같은 리뷰 세션에서 다음과 같은 권고도 함께 나왔습니다.

> **L57** `for _, posting := range postings` — [Rule 31] range 값 원소는 복제본
> - 근거: `model.JobPosting` 구조체의 크기가 클 경우, 단순히 식별자만 읽는 순회라도 매 반복마다 불필요한 값 전체 복제 비용이 발생함.
> - 제안: `for i := range postings { ... postings[i].SourceKey ... }` 형태로 인덱스 접근 사용.

문제는 다음 세 가지입니다.

- Rule 31의 본문(`docs/Gosu_todo.md`)은 *"구조체를 변경하려면"* 인덱스 접근이 필요하다는 것이 핵심이었으나, 이번 리뷰는 *"단순 식별자만 읽는 순회"* 에까지 일반화해 적용했습니다.
- 컴파일러 최적화나 escape analysis 맥락이 빠져 있어, 사실상 영향이 없는 케이스에도 비용이 발생하는 것처럼 단정했습니다.
- Go 1.22에서 `range` 변수의 스코프가 매 반복마다 새로 만들어지도록 바뀐 것 같은 최신 변경사항도 고려되지 않았습니다.

원인 가설은 **LLM의 학습 데이터 컷오프**입니다. 최신 컴파일러 동작·1.22+ 변경 등이 반영되지 않은 채, 옛 Go 입문서에 자주 나오는 권고 패턴을 그대로 일반화해 적용한 것으로 보입니다.

### 해결 방향 메모

- 룰 본문에 **"이 룰이 적용되지 않는 경우(예외)"** 를 명시적으로 적어 라우터가 과적용하지 않도록 합니다.
- 룰 참조에 **Go 버전 컷오프** (예: "Go 1.22 이후 변경된 동작은 ~")를 부속 메모로 두고, 리뷰 출력에서 룰 적용 시 이 메모를 같이 인용하도록 강제합니다.
- 4주차에 룰을 손볼 때 위 두 가지를 우선 적용 대상으로 끌고 갑니다.

## 이번 주차에서 얻은 교훈

- **"AI를 어디에 쓸지"보다 "AI를 *어디에 쓰지 않을지*"가 더 큰 결정**입니다. 회사명 정규화는 LLM으로도 풀 수 있는 문제였지만, 풀 *필요가 없는* 문제로 재정의하는 편이 더 강력했습니다. LLM 단계를 줄이는 것보다 *제거* 하는 쪽이 최선의 최적화일 수 있습니다.
- **"LLM을 쓸까"보다 앞서는 질문이 "다뤄야 할 결정론적 문제를 코드로 감당할 만큼 좁힐 수 있는가"입니다.** 좁힐 수 있으면 코드로 끝, 좁힐 수 없으면(또는 좁혀도 너무 많으면) LLM. 이 순서가 1주차의 범용 크롤러 폐기와 3주차의 회사명 정규화 폐기를 *같은 결정* 으로 묶어줍니다.
- **2주차 산출물은 3주차에 즉시 부담스러운 노이즈를 만들 수 있습니다.** `gosu-review`가 잘못된 옛 Go 정보를 권고한 사례에서, 룰 본문이 *예외 조건을 빠뜨린 채* 일반화되어 있을 때 어떻게 오작동하는지 확인했습니다. 룰 정비가 4주차의 우선 작업으로 들어옵니다.
- **계획과 실제의 격차도 산출물입니다.** 추천 에이전트 적용을 미룬 결정의 이유(데이터 기반이 먼저 필요)가 5주차 마무리에서 회고할 핵심 의사결정 중 하나가 됩니다.
