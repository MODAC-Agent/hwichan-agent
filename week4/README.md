## 3주차 계획과 4주차 실제 진행의 격차

3주차 마지막에 4주차 숙제로 적었던 4개 항목과 실제 결과는 다음과 같습니다.

| 3주차에서 약속한 것 | 4주차 결과 |
|---|---|
| 데이터 정비 마무리 (텍스트/이미지 분기) | ✅ 완료. extractor의 본문 구조화 + OCR backend를 gemini-cli로 일원화 |
| 추천 에이전트 적용 | ❌ 또 미룸. 본문 정제 결과물의 신뢰성을 먼저 잡아야 했음 |
| `gosu-review` 룰 정비 (예외 조건 + Go 버전 컷오프) | ⏭️ 일정 사정으로 5주차 이월 |
| 다른 비정형 입력에 판단 규칙 재적용 | 5주차 추천 작업과 함께 진행 |

추천 파이프라인이 두 주 연속 미뤄진 점은 짚어두려고 합니다. 3주차에는 매칭에 쓸 데이터의 형태부터 좁히느라 미뤘고, 4주차에는 그 결과물을 만드는 LLM 정제 단계와 신뢰성 보강을 먼저 했습니다. 두 번 다 데이터 쪽을 먼저 안정시키자는 같은 판단이 만든 결과인데, 이 부분은 5주차 마무리 회고에서 다시 정리할 예정입니다.

## 1주차 「비정형 데이터 정제」 패턴의 실제 적용

이번 주차의 메인 콘텐츠입니다. 1주차에서 *"AI의 가장 큰 강점은 비정형 데이터 처리에 있다"* 라며 정리한 패턴이 있었습니다 — 형식이 제각각인 입력을 LLM에 맡겨 정형 데이터로 만들어 두면, 그다음 단계는 평범한 코드로 자동화하기가 쉬워진다는 이야기였습니다. 4주차에 와서야 이 패턴을 처음으로 실제 시스템에 적용했습니다. 적용한 곳은 두 군데(채용 본문 텍스트, 이미지)고, 둘 다 매칭이 그대로 받아 쓸 수 있는 정형 데이터를 만드는 *입력 단계*에 들어갑니다.

### extractor: 채용 본문을 매칭 키로 변환

크롤러가 수집해 둔 채용 본문 텍스트는 회사·직무마다 분량과 구조가 모두 다른 전형적인 비정형 데이터입니다. extractor는 이 본문을 받아 *지원자에게 요구되는 항목*을 세 카테고리의 짧은 문장형 배열로 분해해 정형 데이터로 만듭니다.

```text
- "experience":  요구/우대 경력      (예: "Unity 3년 이상")
- "competency": 보유 기술/스킬/도구  (예: "C# 숙련", "Git 협업")
- "trait":       인성/태도/성향       (예: "주도적인 문제 해결")
```

호출 방식은 단순합니다. 각 게시물 본문을 임시 파일에 적은 뒤 `gemini -p <prompt> @<body_file> -o text`로 `gemini-2.5-pro`를 띄워 처리하고, 결과를 `job_posting_extractions` 테이블에 한 게시물당 한 행으로 저장합니다. 매칭 단계가 별도 가공 없이 그대로 가져다 쓸 수 있는 형태입니다 (`5517e7d`, `1c95fe3`).

여기서 핵심은 *프롬프트가 매칭이 받을 입력의 형태를 미리 정해 둔다* 는 점입니다. *"본문을 분류해 줘"* 가 아니라 *"이 세 키를 가진 JSON 객체만 내라"* 라는 식으로 짜 두면, LLM 출력이 매번 다른 모양으로 나올 여지를 매칭 코드가 그대로 받을 수 있는 형태까지 좁힐 수 있습니다. 덕분에 매칭 코드는 LLM 응답을 다시 해석하는 단계 없이 바로 정형 데이터를 다루게 됩니다.

본문 수집 단계도 이번 주차에 한 번 손봤습니다. gamejob 상세 페이지가 본문을 두 iframe(회사 소개·담당업무 / 자격요건·우대사항·전형·근무지)에 나눠 담는다는 점을 뒤늦게 알게 됐고, 두 iframe을 합쳐 한 본문으로 저장하도록 detail scraper를 고쳤습니다 (`68e040b`). LLM에 들어가는 입력의 누락을 줄인 작업이라 정제 결과의 정확도에 직접 영향을 줍니다.

### OCR: 이미지 본문을 텍스트로 변환

채용 공고 중에는 본문이 이미지로 들어 있는 경우가 있습니다. 텍스트 본문이 없는 게시물은 OCR 워커가 이미지를 읽어 텍스트로 바꾼 뒤, `[OCR]` 마커와 함께 본문에 합쳐 둡니다.

이번 주차에 OCR 쪽에서 결정한 두 가지를 정리합니다.

- **PaddleOCR을 떼고 gemini-cli로 통일** (`5f455a2`). PaddleOCR은 별도 Python 런타임과 모델 파일을 따로 관리해야 했는데, gemini-cli는 extractor가 이미 같은 방식으로 부르고 있던 도구라 외부 의존성을 하나로 줄일 수 있었습니다. 결과적으로 두 LLM 사용처(본문 정제·OCR)가 같은 방식으로 호출됩니다.
- **Python 워커 → Go 이식** (`1c95fe3`, `96d12ad`). OCR 워커를 추천 시스템 본체와 같은 코드베이스에 옮겼더니, DB 접근 코드와 모델 정의를 그대로 재사용할 수 있게 됐습니다. 옮기기 전에 Python 쪽 의존성 관리(`uv`)를 한 번 정리하고, Go 워커가 같은 동작을 그대로 따라가도록 옮겼습니다.

OCR도 결과적으로는 *"비정형(이미지) → 정형(텍스트) → 본문 합본"* 흐름이라 extractor와 같은 「비정형 데이터 정제」 패턴의 두 번째 적용 사례입니다.

### 두 적용처가 시스템에 남긴 결과

두 적용처를 정리해 보면, 4주차까지의 시스템에서 LLM이 쓰인 곳은 *입력 단계*의 두 군데입니다 — extractor의 본문 정제, OCR의 이미지 → 텍스트 변환. 두 곳 다 형식이 제각각인 입력을 매칭이 그대로 쓸 수 있는 형태(JSON 배열·텍스트)로 좁혀 주는 역할이고, 결과는 같은 데이터베이스 테이블(`job_posting_extractions`, `job_posting_bodies`)에 저장됩니다.

이 결과로 *매칭이 받을 입력의 형태와 품질이 LLM 정제 단계에서 사실상 결정되는 구조*가 됐습니다. 정제가 잘 안 되면 매칭도 그만큼만 잘 되는 구조라는 의미고, 이번 주차에 안정성 작업을 매칭 쪽이 아니라 정제 단계부터 챙긴 이유이기도 합니다.

매칭 단계에 LLM을 또 둘지(예: 결과 순위를 다시 매기는 단계, 사용자가 자유롭게 입력한 키워드를 해석하는 단계, 본문과의 유사도를 보정하는 단계)는 5주차에 매칭 초안을 설계하면서 같이 검토할 일입니다. 이번 주차에 분명히 정해 둘 수 있는 건 *입력 단계의 두 군데* 까지고, 그다음 매칭 단계에 LLM을 둘지 여부는 정제 결과를 실제 매칭에 한 번 넣어 본 뒤에 판단하는 게 맞다고 봅니다.

## 신뢰성 베이스라인: LLM이 들어갈 자리가 없는 영역

본문 정제 파이프라인을 갖추고 나서, 그다음으로 한 일은 *결과물을 믿을 수 있게 만드는 작업* 이었습니다. 크롤러·extractor·ocr-worker 세 배치 도구에 재시도·타임아웃·병렬화를 같은 방식으로 적용했습니다 (`b9c2015`).

- HTTP 호출은 시도마다 15초 타임아웃을 걸고 `retry.Do`(지수 백오프 + 지터, 3회)로 감쌌습니다. 408/429/5xx와 네트워크 에러만 재시도하고, 그 외 4xx는 바로 실패시킵니다.
- gemini 서브프로세스 호출은 종료 코드 1/124/130과 `DeadlineExceeded`만 재시도합니다 (2회).
- extractor와 ocr-worker의 순차 루프를 `errgroup.SetLimit`으로 워커 풀(기본 3)로 바꿨습니다. 한 게시물에서 gemini 호출이 멈춰도 배치 전체를 막지 않도록 한 변경입니다.

이 영역 전체가 3주차에서 정리한 *"종류가 코드로 감당할 만큼 적게 추려진다 → 코드"* 라는 판단에 그대로 들어맞는 사례입니다. 재시도 정책·동시 실행 수·타임아웃은 LLM의 유연함이 도와줄 문제가 아니라 정해진 규칙으로 처리되는 영역이고, 코드로 끝납니다.

한 가지는 짚어두려고 합니다. **gemini 호출 자체가 잠깐 실패한 경우(네트워크·타임아웃 등)는 재시도하지만, 응답 JSON 파싱이 실패한 경우는 일부러 재시도하지 않습니다.** 모델이 형식이 깨진 응답을 한 번 냈다면 같은 입력으로 다시 부른다고 결과 형식이 크게 달라질 가능성이 낮고, 그 게시물은 다음 배치 실행 때 어차피 다시 처리됩니다. 즉 *환경 쪽 일시적인 실패는 그 자리에서 재시도, 모델 출력 형식이 깨진 경우는 다음 배치에 맡긴다* 는 두 정책을 같은 retry 헬퍼 안에서 다르게 적용했습니다.

## 시스템 전체 개요와 5주차로 남은 빈 칸

지금까지의 누적 작업을 한 도식으로 정리합니다. 세 배치 워커(crawler / ocr-worker / extractor)는 외부에서 순서를 강제받지 않고, 본문 행의 `ready_for_llm` 플래그로 자동 게이팅됩니다.

```
[gamejob 크롤러]
  │  HTML 본문 → job_posting_bodies
  │    · 이미지 없음 → ready_for_llm = true
  │    · 이미지 있음 → ready_for_llm = false
  │  이미지 URL → job_posting_images
  ▼
[OCR 워커 (LLM)]
  이미지가 있는데 OCR 본문이 아직 없는 게시물만 대상
  HTML 본문 + "[OCR]" 마커 + OCR 텍스트로 합쳐 저장
  → ready_for_llm = true 로 플립
  ▼
job_posting_bodies  (ready_for_llm = true 인 행만 다음 단계로)
  ▼
[extractor (LLM)]
  ready_for_llm = true 이고
  추출이 없거나 body.updated_at 이 이전 추출 이후로 갱신된 행 대상
  본문 → experience / competency / trait JSON
  ▼
[구조화된 매칭 입력]
  + experience / competency / trait    (4주차 extractor 출력)
  + 직무 코드 (다중)                     (3주차)
  + 정규화된 회사 ID                     (3주차)
  + 사용자 선호도·경력 연차              (3주차)
  ▼
[추천 매칭]   ← 5주차에서 채울 빈 칸
  ▼
[사용자 추천 결과]
```

이미지가 없는 게시물은 크롤러가 저장하는 시점부터 `ready_for_llm = true`라 OCR을 건너뛰고 바로 extractor 대상이 됩니다. 이미지가 있는 게시물은 크롤러 시점엔 `false`로 두었다가, OCR이 본문을 채우면서 `true`로 바꿔 둔 뒤에야 extractor가 처리합니다. OCR 처리가 나중에 들어와도 `body.updated_at` 갱신을 보고 extractor가 다음 실행 때 다시 처리해 주기 때문에, 세 워커를 동시에 돌리거나 순서를 바꿔 돌려도 결과가 같습니다.

도식 가장 아래의 매칭 단계가 5주차로 남은 빈 칸입니다. 4주차까지의 작업은 매칭이 받아 쓸 입력 데이터의 형태를 정리하는 과정이었고, 매칭 단계 자체를 어떻게 짤지(단순 비교만으로 갈지 / 그 위에 LLM을 더 둘지 / 다른 방식이 있을지)는 5주차에 결정합니다.

## 이번 주차에서 얻은 교훈

- **AI 활용이 적은 주차가 아니라, 1주차에서 정리한 「비정형 데이터 정제」 패턴을 시스템 두 곳(본문·이미지)에 실제로 적용한 주차였습니다.** 두 사례 모두 같은 도구(gemini-cli)를 같은 방식으로 호출하고, 재시도·타임아웃 같은 안정성 작업도 똑같이 적용됩니다. 덕분에 입력 단계에서 LLM을 쓰는 방식이 두 곳 모두 같은 모양으로 통일됐습니다.
- **매칭에 들어갈 입력은 LLM이 만들지만, 매칭 단계 자체에서 LLM을 또 쓸지는 5주차에 설계하며 정할 일입니다.** 입력 정제를 잘 갖췄다고 해서 매칭도 무조건 코드만으로 풀어야 한다는 뜻은 아닙니다. 결과 순위를 다시 매기는 단계나 사용자가 자유롭게 적은 키워드를 해석하는 단계 같은 곳에는 LLM이 들어갈 수 있습니다. 이번 주차에 분명히 정해 둔 건 *입력 단계의 두 군데* 까지입니다.
- **추천 파이프라인이 두 번 연속으로 미뤄진 건, 데이터 쪽부터 안정시키자는 같은 판단을 두 주에 걸쳐 두 번 반복한 결과였습니다.** 3주차엔 매칭에 쓸 데이터의 형태를 좁히는 일을, 4주차엔 그 결과물의 안정성을 먼저 챙겼습니다.
- **외부 의존성은 줄일 수 있을 때 줄이는 게 낫습니다.** PaddleOCR을 떼고 OCR 워커를 Go로 옮긴 결정은, 운영 비용을 줄이는 효과보다 두 LLM 사용처가 같은 방식으로 호출되도록 단순해진 효과가 더 컸습니다. 안정성 작업을 한 번에 두 곳에 적용할 수 있었던 것도 이 정리 덕분입니다.
